# 战略转向：为什么 AI Coding 团队必须做 SDD
> 当 AI 开始参与编码，团队最稀缺的资源不再是“写代码的人力”，而是“稳定、清晰、可传递的上下文”。

## 从个人 Prompt，升级为团队工作流

### 🧭 AI Coding 的真正瓶颈
- 单人使用 AI 时，Prompt 可以临场发挥，但团队协作不能依赖个人记忆
- 一旦需求、边界、验收口径没有被明确记录，AI 每次生成都可能偏向不同方向
- 所谓 SDD，本质上是在实现前先把上下文结构化，避免后续返工失控

### ⚠️ 没有规格时常见的工程后果
[tags]
- [orange] 功能越做越多，但没人能清楚说明边界
- [purple] AI 修了 A，又悄悄改坏 B，变更难以追踪
- [blue] Code Review 只能看代码，无法判断是否符合原始目标
- [green] 引入 OpenSpec 后，团队开始围绕同一套工件协作
[/tags]

### ✅ 哪些项目最适合先导入 SDD
- [x] 需求复杂且会频繁调整的后台系统
- [x] 有权限、审计、状态机、审批流等业务约束的系统
- [x] 多人并行开发、需要分模块协作的项目
- [x] 希望让 AI 输出进入正式研发流程的团队

> **AI 提升的是实现速度，不会自动提升决策质量。**
> 如果没有规格约束，团队只是更快地产生更多需要返工的代码。

# OpenSpec 框架：把模糊需求转成可执行工件
> OpenSpec 的价值，不是多写几份文档，而是让需求、设计、任务、验收之间建立明确映射关系。

## OpenSpec 的核心输出

### 📦 四类关键工件
[flow]
1. proposal.md — 先定义问题、目标、成功标准与非目标
2. design.md — 记录方案选择、系统边界、风险与回滚策略
3. specs/ — 将功能拆成可评审、可测试、可交付的规格单元
4. task.md — 把规格落成开发任务、依赖关系与完成定义
[/flow]

### 🔍 proposal.md 的关键作用
- 让团队先对“为什么做”形成共识，而不是直接争论实现方式
- 约束 MVP 边界，避免需求在实现中不断膨胀
- 把成功标准前置，减少上线后才发现理解不一致

### 🏗️ design.md 应该覆盖什么
- 系统边界：模块分工、数据流向、外部依赖
- 技术选择：为什么选这个方案，为什么不选其他方案
- 风险与回滚：哪些地方最容易出问题，出问题后如何止损

### 🧩 specs/ 为什么是 AI Coding 的关键接口
- AI 最怕的不是复杂任务，而是目标不清的任务
- specs/ 让每一轮实现都基于明确功能边界和验收标准
- 当规格拆分足够清楚时，AI 生成结果更稳定，也更容易审查

```prompt [label="Terminal"]
npm install -g @fission-ai/openspec@latest
openspec init

```


# 实战落地：如何让 AI 按规格稳定交付
> 有效的 AI 开发，不是“一次性生成全部系统”，而是“让每一步输出都有前置规格与后置验证”。

## 一条可执行的交付路径

### 🔄 从需求澄清到上线回写
[flow]
1. 先澄清需求 — 补齐目标、约束、失败路径、验收标准
2. 冻结 MVP 规格 — 先定义第一版必须完成什么
3. 拆分 specs/ — 把复杂系统切成可以单独实现的功能单元
4. 让 AI 按 task.md 实现 — 每次只处理一个任务或一个规格单元
5. 用测试验证 — 将验收标准转成可执行测试场景
6. 上线后回写 — 把偏差、例外、经验补回 design 与 specs
[/flow]

### ✍️ 写 Prompt 时最容易踩的坑
[tags]
- [orange] 一次塞完整系统需求，导致输出缺乏层次和边界
- [purple] 只描述功能界面，不描述异常流程与失败场景
- [blue] 忽略非目标，AI 会主动补出不该做的功能
- [green] 正确做法：每次 Prompt 绑定一个规格单元和明确验收目标
[/tags]

### 🧪 怎样把“验收标准”变成工程约束
- 每条规格都应该能映射到至少一组测试场景，而不是只停留在自然语言
- PR 必须说明对应规格、验证方式和潜在风险，不能只有“代码完成”
- CI 校验的不是“能不能跑”，而是“是否满足规格约束”

> **AI 会放大现有流程。**
> 如果你的流程已经规格化，AI 会放大效率；如果流程混乱，AI 只会放大混乱。

# 团队治理：把 SDD 变成可复用的工程资产
> OpenSpec 最终要解决的，不是某一次开发顺不顺，而是团队是否能持续、稳定地复用同一套协作方法。

## 建议的治理方式

### 🧱 让规格进入日常研发活动
- 需求评审先看 proposal，避免讨论在错误问题上发散
- 方案评审基于 design，确保风险与回滚被提前看到
- 开发任务由 specs/ 和 task.md 驱动，减少上下文切换成本
- 上线复盘反哺规格模板，形成持续进化的团队方法论

### 🔀 多人并行开发时的约束原则
- 一个开发分支尽量只对应一个规格单元，降低冲突范围
- 一个 PR 必须能追溯到 proposal、design 或 specs 中的具体条目
- 高风险功能必须提供验证证据、观测指标与回滚路径

### 📊 建议长期追踪的指标
[summary]
- 🎯 **需求一次通过率** | 从规格评审到开发启动，不因定义不清而返工的比例
- 🧪 **验收对齐率** | 测试用例与规格验收标准的一致性覆盖程度
- ⚙️ **变更稳定度** | 发布后的缺陷率、回滚率与修复时长
- 📚 **规格复用率** | 后续项目直接复用历史规格、模板与 Prompt 的比例
[/summary]

### 🎤 给团队的行动建议
- 先选一个中等复杂度功能试点，不要一开始就覆盖整个系统
- 优先打通 proposal、design、specs、task 这四类核心工件
- 把“规格先行、实现后置、验证闭环”变成团队默认习惯

[summary]
- 🧭 **先定义，再生成** | SDD 的本质是先稳定上下文，再放大实现速度
- 🏗️ **OpenSpec 是团队接口** | proposal、design、specs、task 构成协作主干
- 🔄 **验证与回写不可省略** | 没有验证的规格只是文档，没有回写的经验无法复用
- 🚀 **目标不是写得更快** | 而是让 AI 交付更稳、更可审计、更可协作
[/summary]
