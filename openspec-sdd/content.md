# 开场定位：在 AI Coding 时代重建交付秩序
> AI 把编码速度拉满之后，团队最大的风险不再是“写不出来”，而是“写出来却对不齐、审不动、改不稳”。

## 为什么现在必须做 SDD

### 🚩 典型失控信号
- 功能上线前看似完成，上线后却频繁返工
- PR 数量变多，但评审质量下降，争议集中在目标不清
- 新成员加入后无法理解历史决策，只能靠口头传承

### ⚠️ 不做规格化的真实代价
[tags]
- [orange] 需求漂移：同一能力在不同迭代里定义不断变化
- [purple] 实现漂移：AI 按局部上下文输出，跨模块污染严重
- [blue] 验收漂移：产品、开发、测试对“完成”标准不一致
- [green] 破局方式：OpenSpec + SDD 建立统一交付语义
[/tags]

### ✅ 导入 SDD 的最佳时机
- [x] 你开始频繁使用 AI 参与需求实现
- [x] 项目需要多人并行、跨角色协同
- [x] 业务规则复杂，必须可追溯与可审计
- [x] 团队希望沉淀可复用流程，而不是一次性产出

> **先把问题定义清楚，再把实现速度拉满。**

# OpenSpec 方法：把讨论变成工件，把工件变成执行
> OpenSpec 不是“写文档工具”，而是把需求、设计、任务、验收串成同一条交付链路。

## 核心工件地图

### 🗺️ 从目标到交付的四步
[flow]
1. proposal.md — 明确目标、范围、非目标、成功指标
2. design.md — 记录架构边界、技术权衡、风险与回滚
3. specs/ — 按能力拆分可实现、可验证的规格单元
4. task.md — 把规格转为任务清单、优先级与依赖关系
[/flow]

### 🔍 proposal 的价值：先统一问题定义
- 让团队围绕“为什么做”达成一致，而不是先争实现
- 约束 MVP 边界，防止需求在开发中膨胀
- 通过成功指标把“感觉完成”变成“可判断完成”

### 🏗️ design 的价值：提前暴露风险
- 明确系统边界、外部依赖、关键数据流
- 说明方案取舍依据，避免“后来没人记得为什么这样做”
- 高风险模块必须有降级方案和回滚路径

### 🧩 specs + task 的价值：驱动 AI 稳定执行
- specs 负责把大需求切成可交付单元
- task 负责把规格落到执行顺序和依赖约束
- AI 按规格单元执行，结果更稳定、评审更清晰

# 落地执行：把 AI 变成可治理的产能
> AI 不是“自动交付按钮”，而是“在规则内放大执行力”的引擎。

## 建议流程（可直接套用）

### 🔄 一轮迭代的标准节奏
[flow]
1. 需求澄清会 — 对齐目标、边界、失败场景、验收标准
2. 规格冻结点 — 先锁 MVP，再推进实现
3. 拆分规格单元 — 每个单元对应一组任务和验收项
4. AI 辅助实现 — 每次只处理一个规格单元
5. PR 评审 — 必须绑定规格条目与验证证据
6. CI 验证与上线回写 — 将偏差回写到 specs/design
[/flow]

### 🧠 Prompt 与评审配套规范
[tags]
- [orange] 禁止：一次性让 AI 生成完整系统
- [purple] 禁止：只描述页面，不描述异常与边界
- [blue] 必须：每轮实现都引用具体 specs 条目
- [green] 必须：PR 附上验收证据与风险说明
[/tags]

### 🧪 测试如何跟规格对齐
- 每个规格单元至少对应一组自动化测试场景
- 验收标准必须可执行，不接受模糊描述
- CI 关注“是否满足规格”，而不只是“能否运行”

> **流程不清时，AI 放大混乱；流程清晰时，AI 放大效率。**

# 团队治理：把 SDD 做成可复用能力
> 真正的竞争力不是某次项目交付，而是团队能否持续复用同一套方法稳定产出。

## 组织级实践建议

### 🧱 研发活动与工件绑定
- 需求评审绑定 proposal，减少方向误判
- 方案评审绑定 design，提前处理高风险决策
- 开发与测试绑定 specs/task，统一执行口径
- 复盘绑定回写机制，持续优化模板与流程

### 🔀 并行开发约束
- 一个分支对应一个规格单元，缩小冲突面
- 一个 PR 对应一个明确规格目标，便于审计
- 高风险能力上线前必须完成回滚演练

### 📊 推荐长期跟踪指标
[summary]
- 🎯 **规格一次通过率** | 规格评审后无需返工即可进入开发的比例
- 🧪 **验收覆盖率** | 规格验收项被自动化测试覆盖的程度
- ⚙️ **变更稳定度** | 缺陷率、回滚率、修复时长
- 📚 **模板复用率** | 历史规格与 Prompt 模板在新项目复用占比
[/summary]

### 🎤 结尾行动清单
- 先选一个中等复杂功能作为试点
- 先跑通 proposal、design、specs、task 四类工件
- 用一次真实迭代验证流程，再扩展到全团队

[summary]
- 🧭 **核心原则** | 先定义，再实现；先验收，再上线
- 🏗️ **核心抓手** | OpenSpec 统一需求、设计、任务与验证
- 🔄 **核心能力** | 回写偏差，持续迭代流程模板
- 🚀 **核心目标** | 更可复用、更可审计、更可协作的稳态交付
[/summary]
