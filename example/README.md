### 在终端生成网页

这里我先贴上一段 Prompt，请它帮我生成一个教学讲义网页

```
请扮演熟悉 AI 主题授课的讲师，以「Agent Skills」为主题设计讲义
讲义用纯 html 网页（CSS/JS 都放里面）呈现

下面分享提示词的几个重点：

2. 风格：接着我说明自己想要的风格，如果没想法也可以让 AI 扮演设计师帮你想
3. 架构：最后则是分阶段提出我希望的网页架构，包含开头页面如何呈现、网页层级如何设计、社群链接要放在最后、侧边栏要有目录这些

### 痛点：细节调整困难

但如果你是追求细节的人，很快就会遇到问题。

因为 AI 生成的描述可能不符合你的期待，段落结构也未必是你要的。

此时我们有两个选择：

1. 第一，与 AI 对话调整细节
2. 第二，直接修改 html 网页

但第一种方案随机性太高，还可能会动到不该动的；而第二种方案，就算我本身是工程师，改起来都觉得痛苦。

假使在后续要花很大的力气才能调整，那么初始版本生成速度再快、画面再漂亮；也是很难应用到现实情境的

所以，我们不妨换一种思路

既然生成式 AI 的输出注定充满随机性，那有什么方式，可以在结果生成后，轻松调整细节

我的答案是“建立模板”

因为今天分享的内容包含建立稳定模板，以及如何把它封装为可重复使用的 Agent Skill，所以视频时间较长，还没订阅频道的朋友，可以先订阅一下，这样之后才找得到哦！

## 用 Vibe Coding 建立可重复使用的模板

模板的核心概念，就是不要让 AI 每次都从零开始生成结果，而是让它帮你建立一套“模板”，之后你只要放进不同的内容，就能产出稳定、一致的成果。

这里我就拿自己当案例，受邀到企业内训时，我每次都需要准备课程讲义。假使都用 Vibe Coding 生成讲义，那风格和质量都不可控；但如果有一套模板，那只要替换内容，就能快速产出符合我需求的讲义。

不管你是做简报、写文档，还是建立网页，只要是重复性的工作，这个概念都可以适用。

这里我做了一个尝试，在对话窗口中跟 AI 说

```
我希望刚刚生成出来的网页，是可以通过 Markdown 格式编辑内容的
请帮我将网页结构拆解（列点、步骤、图片、表格、注释）、分层次（大标、中标、小标），并以常见的 Markdown 语法来设计规则
最后建立一个脚本，我只要编辑 Makdown 文件，网页的 HTML 就会及时改变
操作方式写在 README.md 文件中，并建立 package.json
```

期待目标：我希望刚刚生成出来的网页，是可以通过 Markdown 格式编辑内容的
设计架构：请帮我将网页结构拆解（列点、步骤、图片、表格、注释）、分层次（大标、中标、小标），并以常见的 Markdown 语法来设计规则
渲染方式：最后建立一个脚本，我只要编辑 Makdown 文件，网页的 HTML 就会及时改变
说明文件：操作方式写在 README.md 文件中，并建立 package.json

这段提示词，目标是把 HTML 逆向回 Markdown 语法结构。

因为想要对内容有掌控力，就应该把它设计在一个可编辑、方便维护的格式中，而不是放在最终输出的 HTML 里面。

而且我不只是把它变成 Markdown 而已，我还要求它建立一套可以执行的脚本

这样我只要编辑 Markdown 文件，HTML 的内容就会及时调整。

稍等一段时间后，AI 就初步帮我们把模板建立完成了

我们就按照它的指示，输入 `npm run dev`，打开对应的网址

这里可以看到网页有顺利渲染，然后我们再去看一下 AI 设计的 Markdown 文件

调整一下文字，此时你会看到网页有即时更新

接下来我们在终端输入下面的 Prompt，我跟 AI 说：
```
根据下面需求在「.agents」文件夹建立 Skill：
- 参考模板：根据刚刚脚本设计的规格，建立明确的 Markdown 使用规范
- 触发时机：用户提供初步想法想建立网页时，会触发这个 Skill

- 特别提醒：预计会有多个主题，所以触发时，请先建立新的文件夹，里面存放新生成的 Markdown 与 HTML 文件；并且建立独立的 scripts 与 references 文件夹，确保可单独执行

references 文件夹放着 Markdown 格式的参考文件，而 scripts 则是存放模板编译的程序脚本

## 自由度高的背后，是功能的不完善

尽管刚刚设计的 Agent Skill 搭配模板脚本可以轻松生成讲义网页。

但其实功能非常不完善，这里跟大家举例：

1. 首先像这个侧边栏虽然方便导航，但它会占掉讲义展示空间，我希望讲课时可以自由收合
2. 还有目前网页为“浅色”设计，但有时我授课场地是关灯的，如果屏幕是唯一光源，台下观众眼睛看久会不舒服
3. 另外有些单位会希望讲师提供 PDF 讲义，如果使用网页默认的 PDF 打印，那画面会非常丑

这里我只是举出几个常见痛点，虽然建立模板听起来很厉害，但自由的背后，代表你是目前唯一的维护者

因为模板细节是在实际使用后，你才知道哪里要调整；尽管设计与程序编写可以交给 AI，但方向与美感还是要自己掌握。

这就是自己设计模板的代价。你想要主控权，就得承担定制化背后的复杂度。

老实讲，如果只是要一个在线演示工具，市面上已经有很多现有工具，不一定要走自建模板这条路。

而我选择做这件事，与其说是建立一个更好的方案，不如说是在设计一套最符合自己需求的工具。

但点开来后，你会发现页面不存在，这是因为网页文件存在 example 文件夹底下

我们回到 Code 页面，我们的文件在 example 底下

所以只要在网址后面加上 example 就可以顺利显示啦～

### 专业会让细节更完善

### 共用信息

接着再回到代码编辑器

如果有些信息希望反复利用，其实我们可以把它抽出来，放到 config 文件夹，让它可以在所有页面被共用。

而有些希望根据文章定制化调整的信息，我让大家可以通过 config.yaml 文件来覆盖
